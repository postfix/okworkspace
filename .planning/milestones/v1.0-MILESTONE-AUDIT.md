## Human-Item Resolution — Phase 12 (data-loss concern: dismissed)

The Phase-12 verification was `human_needed` over a live-key E2E and a latent durability worry: *does the suggestion row get resolved before the async commit durably lands?* Investigated by code inspection at this audit:

- `ApplyTagsBatch` (batch.go:172) → `EnqueueCommit` → **`EnqueueAndWait`** (commitjob.go:137) **blocks until the commit lands on disk** before returning. Rows are marked applied (batch.go:179) and `ResolvePending` called (handlers_tagsweep.go:234) only AFTER that. **Order is commit-then-resolve — no data-loss window.**
- The live symptom (200 `applied`, no commit increment) = **worker saturation**: ~126 queued KindTagSuggest jobs sat ahead of the KindCommit job, pushing it past `commitWaitTimeout`, whose fallback (commitjob.go:138) returns success leaving the job **queued**. Jobs are SQLite-backed and re-drain on restart, so the commit still lands. The "lost" live runs killed the server before the queue drained.
- **Disposition:** timing/saturation artifact on already-passing drain tests, not a defect. Residual non-blocking note carried as tech debt (eventual-consistency latency under a saturated worker). `12-VERIFICATION.md` updated to `passed` with the resolution recorded.
