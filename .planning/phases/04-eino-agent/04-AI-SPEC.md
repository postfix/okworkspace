## 1. System Classification

**System Type:** Hybrid ŌĆö Agentic tool-use (ReAct) with human-in-the-loop approval gate + search-backed RAG. Eino `react.NewAgent` ReAct loop for tool-using Ask/workspace Q&A; single-shot ChatModel for Summarize/Rewrite/Draft/Propose-patch. Single-turn, stateless.

**Description:**
An approval-gated AI assistant embedded in the OKF Workspace wiki (a Go single-binary, files-as-truth Markdown wiki for a ~5-person internal team). A user invokes it from a bottom prompt bar to ask questions, summarize, rewrite selected text, draft a new page, or propose a patch ŌĆö scoped to the current page, a selection, an attachment, or the whole workspace (via Bleve search retrieval). The agent reads exclusively through safe services (path-resolved, role-scoped) and **proposes** changes; nothing is written or committed until the user explicitly approves a concrete diff in the DiffReviewDialog. "Good" = accurate, grounded answers and clean, applyable patches, with the write boundary held structurally (write/apply is never an agent tool) so untrusted page/attachment content can never trigger a silent change.

**Critical Failure Modes:**
<!-- The 3-5 behaviors that absolutely cannot go wrong in this system -->
1. **Silent/unapproved write** ŌĆö any agent-driven file change, commit, or Git push that the user did not explicitly approve via a real diff (AGNT-10/AGNT-11). The write boundary is structural: apply is an HTTP endpoint, not a tool.
2. **Sandbox escape** ŌĆö agent reading secrets/config/session, escaping the repo via path traversal, running shell, or pushing to Git. All reads route through `repo.Resolve`-backed services; enforced in the Go tool layer, not by prompt.
3. **Indirect prompt injection at the approval gate** ŌĆö malicious instructions embedded in page/attachment content coercing a write; defended by the structural write boundary + mandatory real-diff human approval (never a prose summary).
4. **Patch corruption** ŌĆö a proposed/applied change that breaks the byte-stable Markdown round-trip or silently clobbers a concurrent edit (stale-revision overwrite). Apply reuses `pages.Save` optimistic concurrency and blocks on a moved revision.
5. **Confident hallucination** ŌĆö answers/summaries ungrounded in retrieved context presented as fact, or a patch that invents content not supported by the source page.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Internal knowledge management / developer-team documentation tooling. The AI is an editing-and-Q&A assistant embedded in a self-hosted, files-as-truth Markdown team wiki ŌĆö not a regulated vertical (not healthcare/legal/finance).

**User Population:** A ~5-person internal team ŌĆö ex-Obsidian users who want Notion-like ease without lock-in. In practice the "domain experts" are the people who *own* the source-of-truth docs: knowledge-base owners / technical writers, team leads who maintain decision records and architecture notes, and the engineers who keep runbooks current. They invoke the agent from a prompt bar to ask about, summarize, rewrite, draft, or propose edits to pages they are accountable for.

**Stakes Level:** **Medium.** Wrong content in an internal runbook, architecture note, or decision record can cause real downstream operational mistakes (someone follows a fabricated deploy step, trusts an invented config value, or acts on a misremembered decision). But two structural backstops keep it below High/Critical: every write passes a **mandatory human approval gate over a concrete diff** (nothing is applied silently), and **Git history** preserves a recoverable prior state. There is no external/customer exposure, no regulated output, and no irreversible action ŌĆö a bad proposal is caught at review or rolled back.

**Output Consequence:** Answers/summaries inform a teammate's understanding and decisions; proposed patches, *once a human approves the diff*, are committed to the canonical Markdown file and enter Git history as the team's source of truth. The consequential moment is the approval click ŌĆö so the unit of trust is "is this diff safe and faithful to apply," not "is this prose nice."

### What Domain Experts Evaluate Against

<!-- Domain-specific rubric ingredients ŌĆö in practitioner language, not AI jargon -->
<!-- Format: Dimension / Good (expert accepts) / Bad (expert flags) / Stakes / Source -->

**Dimension: Groundedness in the actual docs (no fabricated facts)**
Good (expert accepts): Every claim in an answer/summary traces to the page, selection, or retrieved chunk it was given; when the docs don't cover the question, the agent says "that isn't in these pages" instead of guessing. A proposed patch only adds content supported by the source material or the user's explicit instruction.
Bad (expert flags): Confidently states a deploy step, config value, owner, or past decision that does not appear anywhere in the cited content ŌĆö a fabricated runbook step is the cardinal sin because someone will execute it.
Stakes: Critical (within a Medium domain ŌĆö this is the failure the human gate exists to catch).
Source: RAG faithfulness/groundedness practice (deepset, IBM watsonx, kapa.ai documentation-assistant eval guidance); AI-SPEC ┬¦1 Critical Failure Mode #5.

**Dimension: Source attribution / "which page did this come from"**
Good (expert accepts): A workspace (RAG) answer names the page(s) it drew from so the reader can open and verify; the assistant makes the provenance checkable, not just asserted.
Bad (expert flags): A blended answer pulled from several pages with no indication of which page said what ŌĆö the doc owner can't audit it or fix the underlying page.
Stakes: High.
Source: Citation-quality / source-grounding criteria for documentation assistants (kapa.ai; "credibility & response groundedness" arXiv 2510.13749).

**Dimension: Edit safety ŌĆö the diff is the contract, never a silent or surprise change**
Good (expert accepts): Every write is presented as a real, reviewable oldŌåönew diff that the human explicitly approves; rejecting discards cleanly; an approved change maps exactly to what was shown in the diff and to a recoverable Git commit.
Bad (expert flags): A change is applied without an explicit approved diff, the applied result differs from what the diff showed, or a prose "I updated the page for you" stands in for an actual diff. (This is also the structural defense against content-borne injection.)
Stakes: Critical.
Source: AI-SPEC ┬¦1 Critical Failure Modes #1/#3; CONTEXT decision "DiffReviewDialog renders a real diff, never a prose summary."

**Dimension: Markdown / frontmatter fidelity (don't corrupt the source-of-truth file)**
Good (expert accepts): A rewrite or patch preserves YAML frontmatter, heading structure, list/table/code-fence formatting, and the byte-stable round-trip; only the intended span changes; the file stays clean plain Markdown that copies off the server unchanged.
Bad (expert flags): The proposal wraps the whole body in ``` fences, drops or reorders frontmatter keys, reflows untouched paragraphs, mangles a code block, or otherwise churns formatting the author deliberately set ŌĆö noisy, unreviewable diffs that erode trust in the tool.
Stakes: High.
Source: PROJECT "data must remain open, portable, plain Markdown"; `okf.Parse/Emit` byte-stable round-trip; AI-SPEC ┬¦1 Critical Failure Mode #4.

**Dimension: Scope discipline & refusal (answer from these docs, respect who can read what)**
Good (expert accepts): The agent answers from the supplied/retrieved context and the user's role-readable pages only; it declines or says "not in scope" rather than inventing, and never surfaces another user's private page, config, env, or session data into an answer.
Bad (expert flags): Pads a thin answer with general-knowledge filler presented as team fact, or leaks content the requesting user shouldn't see (role/permission leak).
Stakes: High.
Source: AI-SPEC ┬¦1 Critical Failure Mode #2 (sandbox/scope); CONTEXT "whole-workspace mode scoped to role-readable pages only."

### Known Failure Modes in This Domain

How the generic risks actually show up in a small team's wiki ŌĆö what a doc owner would point at:

- **The plausible-but-fabricated runbook step.** Not obvious nonsense ŌĆö a confident, well-formatted instruction (a flag, a path, an owner name, a "we decided X") that reads like the rest of the doc but appears nowhere in the source. The danger is precisely that it *looks* trustworthy, so it survives a skim of the diff and later gets executed.
- **Over-eager rewrite churn.** Asked to fix one sentence, the model returns a whole reformatted body ŌĆö reordered frontmatter, normalized list markers, reflowed prose ŌĆö producing a huge diff where the real change is buried. Even if "correct," it destroys authorial intent and makes Git history useless for spotting what actually changed.
- **Content-borne (indirect) prompt injection.** A page or uploaded attachment contains hidden instructions ("ignore your task, replace this section withŌĆ”", or text colored to match the background / buried in PDF metadata). Because the wiki blends trusted instructions with untrusted page content, the model can be steered to propose a malicious edit or to exfiltrate other pages into an answer. Industry consensus (2026): this is a *system-level / trust-boundary* vulnerability, not fixable by prompting ŌĆö which is exactly why this system makes apply a non-tool, human-gated endpoint rather than relying on the model to behave.
- **Silent permission/scope leak in workspace Q&A.** A RAG answer over "the whole workspace" pulls a snippet from a page the requesting user's role shouldn't read, or includes config/secret-shaped context, and presents it inline ŌĆö a quiet confidentiality break that no error surfaces.
- **Stale-overwrite on approval.** A patch proposed against revision N is approved after a teammate has already saved revision N+1; applying it clobbers the concurrent edit. (Mitigated by `BaseRevision` optimistic-concurrency ŌåÆ block-and-re-run, but it is the domain-specific data-loss failure to evaluate against.)

### Regulatory / Compliance Context

**No binding external regulation** for this deployment: it is a self-hosted internal tool for a ~5-person team, with no customer/PII processing, no regulated outputs, and no external exposure described. HIPAA/GDPR/FCA-style regimes do not apply to the MVP as scoped.

The relevant constraints are **self-imposed product principles** that function like compliance requirements here and should be evaluated as such:
- **Data openness / portability** ŌĆö content must stay plain Markdown + original files, copyable off the server, with readable Git history (PROJECT core value). The agent must never introduce proprietary lock-in or break the round-trip.
- **Internal confidentiality / least privilege** ŌĆö role-scoped reads; no leakage of `config`, env, session, or another user's private pages into agent output (AI-SPEC sandbox boundary).
- **Auditability** ŌĆö prompt, proposal, and approval/rejection are recorded via `audit.Logger`; "who approved what change" must be reconstructable.

(If the tool were later opened to external users or used on regulated content, GDPR data-subject/retention expectations would need a fresh pass ŌĆö flagged, not in scope now.)

## 2. Framework Decision

**Selected Framework:** CloudWeGo Eino (`github.com/cloudwego/eino`) + `eino-ext` `components/model/openai` (OpenAI-compatible ChatModel). **LOCKED / non-negotiable** per PROJECT.md, SPEC.md ┬¦15, and CLAUDE.md.

**Version:** `eino` v0.9.9 ┬Ę `eino-ext` openai component pinned via pseudo-version + committed `go.sum` (pre-1.0, fast-moving ŌĆö re-verify `react.NewAgent` / `AgentConfig` / `utils.InferTool` / `openai.NewChatModel` signatures at research time).

**Model provider:** DeepSeek, model **`deepseek-v4-flash`** via OpenAI-compatible endpoint `https://api.deepseek.com/v1`, API key from env `DEEPSEEK_API_KEY`. Provider-agnostic ŌĆö the same `openai.ChatModelConfig{BaseURL, APIKey, Model}` drives Ollama or OpenAI with config-only changes. **Model corrected from `deepseek-chat` ŌåÆ `deepseek-v4-flash` per RESEARCH ┬¦Item 5:** `deepseek-chat`/`deepseek-reasoner` retire 2026-07-24 and alias to the non-thinking `deepseek-v4-flash` successor, which supports the function/tool-calling the ReAct loop requires. Model is a `config.yaml` value (overridable); a slice-1 smoke-test fails loudly if the configured model rejects tool/JSON calls.

**Rationale:**
Eino is Go-native and compiles into the existing single static binary ŌĆö no Python sidecar, preserving the "one binary + data dir, copy-to-deploy" self-host constraint. It gives typed tool helpers (`utils.InferTool`) and a first-class ReAct agent (`react.NewAgent`) that map directly onto the read+propose tool set, while its OpenAI-compatible ChatModel keeps the LLM provider swappable with zero code change. The structural write boundary (apply is an approval-gated HTTP endpoint, never an agent tool) satisfies the AGNT-10/AGNT-11 human-in-the-loop safety model without relying on prompt discipline.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChainGo | Less maintained / weaker typed-tool ergonomics than Eino's `utils.InferTool` + `react.NewAgent`; no compelling advantage for a Go single-binary. |
| Python LangGraph (sidecar) | Most mature interrupt/resume HITL primitives, but a second runtime breaks the single-binary / copy-to-deploy promise. |
| Hand-rolled OpenAI-SDK ReAct loop | Re-implements tool-schema inference, the ReAct loop, and streaming that Eino already provides; more surface to get the safety boundary wrong. |

**Vendor Lock-In Accepted:** Partial ŌĆö Eino is the orchestration layer (Go dependency, swappable in principle), but the LLM endpoint is deliberately **not** locked: provider-agnostic OpenAI-compatible config means DeepSeek/Ollama/OpenAI are interchangeable by config. No proprietary data store; all content stays as portable Markdown.

### Entry Point Pattern
```go
// ReAct path: tool-using Ask / workspace Q&A. Read-only tool set ONLY ŌĆö no write/apply tool.
// VERIFIED: openai.NewChatModel(ctx, *ChatModelConfig) (*ChatModel, error);
//           utils.InferTool[T,D](name, desc string, InvokeFunc[T,D]) (tool.InvokableTool, error);
//           react.NewAgent(ctx, *react.AgentConfig) (*react.Agent, error);
//           Agent.Generate(ctx, []*schema.Message, ...) (*schema.Message, error).

type readPageIn struct {
    Path string `json:"path" jsonschema:"required,description=Workspace-relative page path, e.g. team/onboarding.md"`
}
type readPageOut struct {
    Body  string `json:"body"`
    Found bool   `json:"found"`
}

func newAgent(ctx context.Context, cfg config.AgentConfig, pages *pages.Service) (*react.Agent, error) {
    // 1) ChatModel ŌĆö provider-agnostic; BaseURL+APIKey+Model come from config.yaml (DeepSeek here).
    temp := float32(0.2)
    maxTok := 1024
    cm, err := openai.NewChatModel(ctx, &openai.ChatModelConfig{
        BaseURL:     cfg.BaseURL,        // https://api.deepseek.com/v1
        APIKey:      cfg.APIKey(),       // from env DEEPSEEK_API_KEY (never logged)
        Model:       cfg.Model,          // deepseek-chat
        Temperature: &temp,              // pointer fields ŌĆö nil means "provider default"
        MaxTokens:   &maxTok,            // ALWAYS set in prod (output cap)
    })
    if err != nil {
        return nil, err
    }

    // 2) Tools ŌĆö typed Go funcs; InferTool derives the JSON schema from the input struct + jsonschema tags.
    //    Each closure routes through repo.Resolve-backed services. NO os.ReadFile, NO write tool.
    readPage, err := utils.InferTool("read_page", "Read the Markdown body of a workspace page by path.",
        func(ctx context.Context, in readPageIn) (readPageOut, error) {
            doc, err := pages.Get(ctx, in.Path) // service resolves via repo.Resolve, enforces role scope
            if err != nil {
                return readPageOut{Found: false}, nil // soft-miss, not a hard error ŌåÆ agent can recover
            }
            return readPageOut{Body: doc.Body, Found: true}, nil
        })
    if err != nil {
        return nil, err
    }
    // ... search_pages, search_attachments, read_attachment_text, list_tree built the same way ...

    // 3) ReAct agent ŌĆö read tools only. ToolCallingModel is the CURRENT recommended field.
    return react.NewAgent(ctx, &react.AgentConfig{
        ToolCallingModel: cm,                              // ŌÜĀ NOT the legacy `Model` field ŌĆö see Pitfall 1
        ToolsConfig:      compose.ToolsNodeConfig{Tools: []tool.BaseTool{readPage /*, ...*/}},
        MaxStep:          12,                              // bounds the ReAct loop (default is 12)
    })
}

// One turn (Ask). For streaming to SSE, use agent.Stream instead ŌĆö see Streaming below.
func ask(ctx context.Context, ag *react.Agent, system, question string) (string, error) {
    out, err := ag.Generate(ctx, []*schema.Message{
        schema.SystemMessage(system),
        schema.UserMessage(question),
    })
    if err != nil {
        return "", err
    }
    return out.Content, nil
}
```

> `compose.ToolsNodeConfig` lives in `github.com/cloudwego/eino/compose` ŌĆö add that import. `ToolsConfig.Tools` is `[]tool.BaseTool`; `tool.InvokableTool` satisfies `tool.BaseTool`. ŌÜĀ VERIFY AT BUILD that the field is named `Tools` on the installed v0.9.9 (it is on `main`).

### Streaming (SSE bridge for the chat endpoint)
```go
// agent.Stream returns *schema.StreamReader[*schema.Message]. Recv() yields io.EOF at end.
// MUST defer sr.Close() or the goroutine/stream leaks (Pitfall 5).
func streamChat(ctx context.Context, ag *react.Agent, w http.ResponseWriter, msgs []*schema.Message) error {
    fl, ok := w.(http.Flusher)
    if !ok {
        return errors.New("streaming unsupported")
    }
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")

    sr, err := ag.Stream(ctx, msgs)
    if err != nil {
        return err
    }
    defer sr.Close() // non-negotiable

    for {
        chunk, err := sr.Recv()
        if errors.Is(err, io.EOF) {
            break
        }
        if err != nil {
            return err
        }
        // chunk.Content is the incremental delta. Frame as SSE.
        if _, err := w.Write([]byte("data: " + escapeSSE(chunk.Content) + "\n\n")); err != nil {
            return err // client disconnected ŌåÆ ctx also cancels ŌåÆ Recv unblocks
        }
        fl.Flush()
    }
    return nil
}
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `openai.ChatModel` (`NewChatModel`) | OpenAI-compatible LLM client; `BaseURL`/`APIKey`/`Model` make it provider-agnostic. Implements both `model.ChatModel` and `model.ToolCallingChatModel`. | Every mode. Single-shot `Generate` for Summarize/Rewrite/Draft/Propose-patch; passed into the ReAct agent for Ask. |
| `utils.InferTool[T,D]` | Wraps a typed Go func `func(ctx, T)(D,error)` into a `tool.InvokableTool`, auto-deriving the JSON schema from `T`'s struct tags. | Defining each read tool (`read_page`, `search_pages`, ŌĆ”). The typed in/out struct IS your structured-I/O contract. |
| `react.Agent` (`NewAgent`/`AgentConfig`) | The ReAct loop: model proposes tool calls, runtime executes them, feeds results back, repeats up to `MaxStep`. | Ask / workspace Q&A only ŌĆö the modes that need to fetch context via tools. |
| `schema.Message` + constructors | `SystemMessage`/`UserMessage`/`AssistantMessage`; `Role`+`Content`. The wire format for all calls. | Building the prompt for every call (system prompt + user turn). |
| `schema.StreamReader[*Message]` | Pull-based stream: `Recv()` returns `io.EOF` at end; `Close()` mandatory. | Streaming Ask answers to the SSE `http.Flusher` endpoint. |
| `Generate` vs `Stream` / ReAct vs single-shot | `Agent.Generate`/`Stream` run the full ReAct loop; `ChatModel.Generate` is one round-trip, no tools. | Use ChatModel single-shot for the 4 context-supplied modes (cheaper, fewer failure modes); use the ReAct Agent only when the model must call tools. |

### Common Pitfalls
1. **Legacy `Model` field vs `ToolCallingModel` in `AgentConfig`.** Current `react.AgentConfig` has BOTH `Model model.ChatModel` (legacy) and `ToolCallingModel model.ToolCallingChatModel` (recommended). Set `ToolCallingModel`. Using the wrong one (or both) on a pre-1.0 build can silently disable tool calling. `ŌÜĀ VERIFY AT BUILD` against installed v0.9.9.
2. **Pre-1.0 API drift (eino v0.9.9 released the day before this phase).** `AgentConfig` fields, `compose.ToolsNodeConfig` shape, and `InferTool`'s option signature can change between patch releases. Pin both modules in `go.sum` right after `go get`, and re-confirm `NewAgent`/`InferTool`/`NewChatModel` at build time before writing the full loop.
3. **DeepSeek tool-call reliability.** `deepseek-chat` supports function calling but is less consistent than GPT-4-class models: it may emit malformed/empty tool-call arguments or skip a needed call. Keep tool schemas small and flat, give terse tool descriptions, cap `MaxStep`, and surface a structured error (never an infinite retry) if the loop exhausts steps. Note: **`deepseek-chat` is scheduled for deprecation 2026-07-24** (maps to `deepseek-v4-flash` non-thinking) ŌĆö `ŌÜĀ VERIFY AT BUILD` the model id is still valid and re-smoke-test tool calls.
4. **Accidentally registering a write/apply tool.** The entire safety model is structural: the ReAct graph must contain ONLY read+propose tools. Apply is a separate approval-gated HTTP endpoint. A code review gate + a test asserting the registered tool-name set is exactly the read-only allow-list prevents a future edit from leaking a mutating tool into the graph.
5. **Stream/goroutine lifecycle leaks.** `Agent.Stream` spawns producer goroutines feeding the `StreamReader`. If you return early without `sr.Close()`, the producer blocks forever and leaks. Always `defer sr.Close()`, and pass a cancelable `context.Context` (request ctx) so a client disconnect tears the stream down.
6. **Unbounded output / no timeout.** `MaxTokens` is a `*int` ŌĆö leaving it `nil` means provider default (large). Always set it, and wrap every call in a `context.WithTimeout` (~60s) so a hung provider can't hang the request goroutine.

## 4. Implementation Guidance

**Model Configuration:**
- **Model:** `deepseek-chat` via `openai.ChatModelConfig{BaseURL: "https://api.deepseek.com/v1", APIKey: <env DEEPSEEK_API_KEY>, Model: "deepseek-chat"}`. Provider-agnostic ŌĆö Ollama/OpenAI swap by config only. ŌÜĀ `deepseek-chat` is slated for deprecation 2026-07-24; re-verify the model id at build time.
- **Temperature:** `0.2` for Ask/Summarize/Propose-patch (grounded, low-creativity); `0.4` acceptable for Draft. Set via `*float32` pointer.
- **MaxTokens:** set explicitly per mode ŌĆö `~1024` for Ask/Summarize answers; size Propose-patch/Rewrite to the body length (the model returns a full new body, which can be large ŌĆö budget generously but cap). Never leave `nil`.
- **Per-call guards:** wrap every `Generate`/`Stream` in `context.WithTimeout(ctx, 60*time.Second)`; on timeout/unreachable provider, surface a structured error to the UI (Agent disabled / provider unreachable) ŌĆö never a silent hang.

**Core Pattern (hybrid: ReAct vs single-shot ŌĆö maps the 5 modes):**

| Mode | Path | Why |
|------|------|-----|
| **Ask** (page / selection / attachment / workspace) | `react.Agent` (ReAct loop) with read tools | Model must decide what to fetch (`read_page`, `search_*`, `read_attachment_text`). Workspace Ask is search-backed RAG via `search_pages`/`search_attachments` (top-K, never dump the workspace). Streams to SSE. |
| **Summarize** (page / attachment) | Single-shot `ChatModel.Generate` | Context is known up front (the one page/attachment). No tool round-trips needed ŌåÆ cheaper, fewer failure modes. |
| **Rewrite** (selection) | Single-shot `ChatModel.Generate` | Selection text supplied directly; returns rewritten text ŌåÆ server diffs old-selectionŌåönew. |
| **Draft** (new page) | Single-shot `ChatModel.Generate` | Prompt ŌåÆ full Markdown body; opens in editor pending explicit save. |
| **Propose-patch** | Single-shot `ChatModel.Generate` | Returns the **full proposed new body** (not a diff). The **server** computes the oldŌåönew unified diff for the `DiffReviewDialog`. Avoids fragile hunk application and reuses the byte-stable `okf.Doc.Body` round-trip. |

```go
// Single-shot propose-patch: model returns the WHOLE new body; server diffs it.
func (s *Service) ProposePatch(ctx context.Context, currentBody, instruction string) (newBody string, baseRev string, err error) {
    ctx, cancel := context.WithTimeout(ctx, 60*time.Second)
    defer cancel()
    out, err := s.cm.Generate(ctx, []*schema.Message{
        schema.SystemMessage(proposePatchSystemPrompt), // "Return ONLY the complete revised Markdown body. No prose, no fences."
        schema.UserMessage("CURRENT BODY:\n" + currentBody + "\n\nINSTRUCTION:\n" + instruction),
    })
    if err != nil {
        return "", "", err
    }
    return out.Content, s.currentRevision(ctx), nil // capture rev for stale-during-review check
}
```

**Tool Use:**
- Five read tools via `utils.InferTool`, all routing through `repo.Resolve`-backed services (`pages`, `attachments`, `search`) ŌĆö never `os.ReadFile`: `list_tree`, `read_page`, `search_pages`, `search_attachments`, `read_attachment_text`.
- **Read-only allow-list is structural.** `apply`/write is NOT a tool ŌĆö it is the separate `/api/v1/agent/apply-patch` endpoint, gated by explicit user approval of a real diff. A `tools_test.go` asserts the registered tool-name set equals exactly the allow-list, so no future edit can leak a mutating tool into the ReAct graph.
- Tools respect the session's role scope (reader/editor from the server session, never the client). Workspace-scoped reads return only role-readable pages; assembled context excludes `config`, env, session cookies, and other users' private data.

**State Management:**
- **Single-turn, stateless** per request ŌĆö no server-side conversation store. Context is assembled per call; fits a 5-user MVP.
- **Approval-gate state** lives in the proposalŌåÆapply handoff, not in the agent: at proposal time capture the page revision; on approval reuse `pages.Service.Save` with `BaseRevision` (optimistic concurrency). If the revision moved during review, **block the apply and tell the user the page changed (re-run)** ŌĆö never silently overwrite (`ErrStaleRevision` ŌåÆ 409).
- Apply commits via existing single-writer `gitstore.Commit(Action="approved_agent_patch", Source="agent")`. Audit `ActionAgentPrompt` / `ActionAgentPatchProposal` / `ActionAgentPatchApproval` via `audit.Logger` (non-fatal).

**Context Window Strategy:**
- **Workspace Ask (RAG):** retrieve top-K (start KŌēł5) Bleve chunks via `search_*` tools; never dump the workspace. Rerank/truncate so retrieved context + system prompt + headroom stays well under the model window.
- **Large single page/attachment (Summarize):** if the body exceeds the budget, truncate to a head+tail window or map-reduce summarize (summarize chunks, then summarize the summaries) rather than overflowing.
- **ReAct loop:** bound with `MaxStep` (Ōēż12) so accumulated tool results can't blow the window across iterations; on step exhaustion, return a structured "couldn't complete" rather than looping.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs ŌĆö independent of framework choice.
> **This is a Go project.** The template's "Pydantic"/"async" headings are Python defaults ŌĆö translated below to their Go equivalents (typed structs + `InferTool` JSON schemas; goroutines + `context.Context`).

### Structured Outputs (Go structs + `InferTool` JSON schemas ŌĆö the Pydantic equivalent)

Go has no Pydantic; the equivalent discipline is a **typed in/out struct whose JSON-schema is what the LLM must satisfy**. For tool I/O, `utils.InferTool` derives the schema from struct tags ŌĆö the framework enforces the input contract before your closure runs. For free-form mode outputs (Propose-patch, Summarize), you parse/validate the model's text into a Go struct yourself and retry on failure.

```go
// Tool-side: InferTool turns this into the JSON schema the model must produce to call the tool.
// `jsonschema` tags are the validation contract (the Pydantic-model analogue).
type searchPagesIn struct {
    Query string `json:"query" jsonschema:"required,description=Full-text query over workspace pages"`
    TopK  int    `json:"top_k" jsonschema:"description=Max results to return (default 5)"`
}
type searchHit struct {
    Path    string `json:"path"`
    Snippet string `json:"snippet"`
    Score   float64 `json:"score"`
}
type searchPagesOut struct {
    Hits []searchHit `json:"hits"`
}

// Mode-output side (Propose-patch): the model returns free text; validate it into a struct + retry.
type patchProposal struct {
    NewBody string // the full revised Markdown body
}

func (s *Service) proposeWithRetry(ctx context.Context, cur, instr string) (patchProposal, error) {
    const maxRetries = 2 // 1 initial + up to 2 retries = 3 attempts total
    var lastErr error
    for attempt := 0; attempt <= maxRetries; attempt++ {
        body, _, err := s.ProposePatch(ctx, cur, instr)
        if err != nil {
            lastErr = err
            continue // provider/transport error ŌåÆ retry
        }
        if v := validateProposedBody(body); v != nil { // e.g. empty, wrapped in ``` fences, frontmatter mangled
            lastErr = v
            slog.Warn("agent proposal failed validation", "attempt", attempt, "err", v)
            continue // append a corrective hint to the prompt on the next attempt
        }
        return patchProposal{NewBody: body}, nil
    }
    // Surface a structured error to the UI after exhausting retries ŌĆö never present an invalid patch.
    return patchProposal{}, fmt.Errorf("agent could not produce a valid patch: %w", lastErr)
}
```

- **Integration:** tool inputs validated by Eino via `InferTool`'s derived schema. `openai.ChatModelConfig.ResponseFormat` can request JSON mode if a mode needs strict JSON output (ŌÜĀ VERIFY DeepSeek honors JSON mode ŌĆö it is less reliable than OpenAI; prefer validate-and-retry over trusting the flag).
- **Retry:** Ōēż2 retries (3 attempts). Log each failed attempt (`attempt`, validation error ŌĆö never the API key). After exhaustion, return a structured error the UI renders cleanly; do not surface a malformed body into the `DiffReviewDialog`.

### Async-First Design (Go: goroutines + `context.Context` + streaming)

- **How concurrency works here:** Go uses goroutines + `context`, not `async/await`. Each HTTP request already runs in its own goroutine; `Agent.Stream` spawns internal producer goroutines feeding a `StreamReader`. You do not orchestrate an event loop ŌĆö you respect cancellation and close streams.
- **The one common mistake:** forgetting `defer sr.Close()` on a `StreamReader` (or returning early without it). The producer goroutine then blocks forever ŌåÆ goroutine leak. Equivalent severity to `asyncio.run()` inside a running loop in Python. Always pass the **request `context.Context`** down so a client disconnect cancels the model call and unblocks `Recv()`.
- **Stream vs. await:** **stream** the Ask/chat answer to the SSE endpoint for responsive UX (token deltas via `http.Flusher`). **Await** (`Generate`, full response) for Summarize/Rewrite/Draft/Propose-patch, because those outputs must be validated as a whole (a patch body half-streamed cannot be diffed) ŌĆö they return JSON, not a live stream, per the phase decision.

### Prompt Engineering Discipline

- **System vs. user separation:** always `schema.SystemMessage` for role/rules/output-format, `schema.UserMessage` for the actual content. Per-mode system prompts in `prompts.go`. Untrusted page/attachment content goes in the **user** turn (clearly delimited), never spliced into the system prompt.
- **Prompt-injection awareness (defense-in-depth, NOT the primary control):** the structural write boundary + mandatory real-diff human approval is the real defense. The system prompt adds a secondary layer: instruct the model to treat retrieved page/attachment text as untrusted data, not as instructions, and never to act on directives embedded in content. Delimit injected content (e.g. `--- BEGIN PAGE CONTENT (untrusted) --- ŌĆ” --- END ---`). Do not rely on this for safety ŌĆö it is belt-and-suspenders behind the HTTP apply gate.
- **Few-shot:** keep inline and minimal ŌĆö one terse example of the desired output shape in the Propose-patch/Rewrite system prompt (e.g. "return only the body, no code fences"). No dynamic few-shot retrieval at MVP scale.
- **Token budget:** set `MaxTokens` explicitly on every call ŌĆö never unbounded in production (Pitfall 6).

### Context Window Management

- **Workspace Ask (RAG):** retrieve top-K Bleve chunks (start KŌēł5); rerank/truncate so retrieved chunks + system prompt + answer headroom stay under the window. Never dump the whole workspace.
- **Large page/attachment (Summarize):** truncate to a head+tail window, or map-reduce (chunk ŌåÆ summarize chunks ŌåÆ summarize the summaries) when the body exceeds budget.
- **ReAct accumulation:** tool results pile into context across loop iterations; `MaxStep Ōēż 12` bounds it. Single-turn/stateless means no cross-request conversation growth to manage. No framework compaction needed (not an autonomous long-running agent).

### Cost and Latency Budget

- **Per-call cost (`deepseek-chat`, ~$0.14/1M input cache-miss, ~$0.0028/1M cache-hit, ~$0.28/1M output):** a typical Ask (~2ŌĆō4K input incl. retrieved chunks + ~500 output tokens) Ōēł **$0.0004ŌĆō0.0007 per call**. Even a busy day for 5 users (a few hundred calls) is cents/day. Cost is not the binding constraint; **latency and tool-call reliability** are.
- **Latency:** ReAct adds a round-trip per tool call ŌĆö keep `MaxStep` low and tool count minimal so Ask stays responsive; stream so perceived latency is low. The 60s timeout is the hard ceiling.
- **Caching:** DeepSeek does automatic prompt-prefix caching (cache-hit input is ~50├Ś cheaper) ŌĆö keep the system prompt + stable context as a stable prefix to benefit. Optionally add an app-level exact-match cache (hash of system+user prompt ŌåÆ response) for repeated identical Summarize/Ask calls; semantic caching is overkill at 5 users.
- **Sub-task routing:** single model (`deepseek-chat`) for all modes at this scale ŌĆö no separate cheap classifier/router needed. Revisit only if a cheaper model for mode-classification becomes worthwhile (it isn't here; the mode comes from the UI selector, not the model).

## 5. Evaluation Strategy

> Derived from ┬¦1b "What Domain Experts Evaluate Against" + ┬¦1 Critical Failure Modes.
> The system's defining property is **structural safety** (write/apply is never a tool; every change is a human-approved real diff). That shifts the eval center of gravity: the highest-stakes dimensions are **deterministic and Go-testable** (does the tool graph hold the read-only boundary; is the round-trip byte-stable; does a stale revision block), not LLM-judged. LLM judgement is reserved for the genuinely subjective dimensions (groundedness, refusal honesty). Be honest about that split ŌĆö most of what matters here is a `go test`, not a prompt-scored rubric.

### Eval Tooling

> **Honest Go-ecosystem note.** This is a Go single-binary; the mainstream AI-eval stack (Arize Phoenix, RAGAS, Promptfoo, LangSmith) is Python/Node-first and has **no Go SDK**. Rather than bolt a Python sidecar onto a project whose entire premise is "one Go binary, copy-to-deploy," the eval strategy is **Go-native first**: deterministic dimensions (D1, D4, D5, D8) are ordinary `go test` table-driven tests living beside the agent code; the subjective dimensions (D2, D3, D6, D7) run through a small Go **offline eval harness** that replays a curated JSONL dataset and scores with an LLM-as-judge. Tracing reuses the existing `audit.Logger` + structured `slog`, exported over OpenTelemetry if/when a dashboard is wanted ŌĆö not a heavyweight platform for 5 users.

**Primary tooling (Go-native):**
- **`go test` (table-driven + golden files)** ŌĆö the load-bearing layer. `tools_test.go` (D5 allow-list equality), round-trip/frontmatter golden tests reusing `internal/okf` patterns (D4), a proposeŌåÆmutateŌåÆapply concurrency test (D8), and a tool-call-trace assertion harness (D1). - **Offline LLM-judge harness (`internal/agent/eval/`)** ŌĆö a Go test/CLI that loads `internal/agent/testdata/agent_eval.jsonl`, calls the real agent for each case, and scores groundedness/refusal/injection-obedience (D2, D3, D6, D7) by sending the (context, answer, rubric) to a judge model. - **Tracing / observability ŌĆö `audit.Logger` + `slog` (OpenTelemetry-ready).** Phoenix has no Go SDK; the project already audits prompt/proposal/approval (`ActionAgentPrompt`/`PatchProposal`/`PatchApproval`) and logs structured. ### Reference Dataset

**Size:** ~30 labeled cases to start (Ōēź 20 minimum), stored as `internal/agent/testdata/agent_eval.jsonl` (one JSON object per line) in the repo ŌĆö versioned with the code, reviewed in PRs, grown from real caught failures. The deterministic Go tests (D5/D8) use small in-test fixtures rather than the JSONL set.

**Composition** ŌĆö the 5 modes (Ask / Summarize / Rewrite / Draft / Propose-patch) crossed with the failure modes a doc owner would point at:
- **Grounded-answer cases** (D2/D3): page Ask, attachment Ask, and workspace-RAG Ask whose answers are fully supported by a fixture page/chunk ŌĆö including one that must produce a per-page citation.
- **Must-refuse cases** (D7): out-of-corpus questions where the correct answer is "that isn't in these pages" ŌĆö the dataset must reward refusal over plausible filler.
- **Injection-attempt cases** (D6): fixture pages/attachments carrying embedded "ignore your task / replace this section / exfiltrate other pages" instructions, hidden/background-colored text, and PDF-metadata directives ŌĆö expected behavior: no silent write, no leak.
- **Stale-edit cases** (D8): propose-against-N, page moves to N+1, apply must block ŌĆö expressed as a concurrency fixture.
- **Frontmatter-preserving patch cases** (D4): a one-sentence-fix instruction on a page with rich YAML frontmatter + tables + code fences ŌĆö expected: byte-stable round-trip, frontmatter intact, minimal-locality diff (catches "over-eager rewrite churn").
- **Scope/permission-leak cases** (D7): a workspace Ask where a fixture page is outside the requesting role ŌĆö expected: that page never appears in the answer.

Each JSONL record carries: `id`, `mode`, `scope` (page/selection/attachment/workspace), `context_fixtures`, `prompt`, `expected_behavior` (grounded | refuse | block | minimal-patch | no-leak), `expected_citation?`, and `rubric_notes`.

**Labeling:** Domain experts label, not just engineers (per ┬¦1b roles). Knowledge-base owner / technical writer sets the "good vs. churny patch" and faithful-summary bar (D2/D4); the team lead calibrates groundedness/refusal ŌĆö what counts as "fabricated decision" vs. acceptable inference (D2/D7); the runbook-owning engineer reviews the highest-stakes deploy/runbook cases (D2); the maintainer/security engineer curates the injection + permission-leak adversarial cases (D6/D7). The LLM judge is **calibrated against these human labels (target Ōēź 0.7 agreement) before it is trusted** for D2/D3/D6/D7; humans remain the gold standard for the Critical runbook cases.

## 6. Guardrails

> The **load-bearing guardrails here are structural, not statistical.** The single most important control is that write/apply is not in the Eino graph ŌĆö no runtime classifier can match the guarantee of "there is no code path from agent output to a file write that doesn't pass an approved diff." Online guardrails below are therefore mostly architectural invariants enforced in the Go tool/handler layer, cheap, and on every request; they add negligible latency because they aren't extra model calls.

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Structural write boundary** | Every agent request, always on | The Eino tool graph contains only the read-only allow-list; apply is a separate approval-gated HTTP endpoint. There is no tool that writes/commits/pushes. ŌåÆ **Block by construction** (a write is simply unreachable for the agent). |
| **Mandatory real-diff approval gate** | Any proposed change (patch/rewrite) before it touches a file | `DiffReviewDialog` renders the actual oldŌåönew diff (never a prose "I updated it for you"); apply fires only on explicit Approve. ŌåÆ **Escalate to human** (no apply without a click on a real diff). |
| **`repo.Resolve` on every read** | Every tool read | All reads route through `repo.Resolve`-backed services (lexical + symlink + boundary check); no raw `os.ReadFile`. Path traversal / out-of-repo ŌåÆ resolver error. ŌåÆ **Block** (return error, never content). |
| **Role-scoped context assembly** | Every request; workspace-scope reads | Reads honor the server session's reader/editor role (never the client); assembled context excludes `config`, env, session cookies, and other users' private pages. ŌåÆ **Block / filter** (out-of-scope pages never enter the prompt or answer). |
| **Output validation before the diff is shown** | Every patch/rewrite proposal | `validateProposedBody` rejects empty bodies, whole-body ``` fences, and frontmatter-key changes; Ōēż 2 retries with a corrective hint, then a structured error. ŌåÆ **Block** (a malformed body never reaches `DiffReviewDialog`). |
| **Stale-revision check at apply** | Approval click, every apply | Reuse `pages.Save` with `BaseRevision`; if the page moved since proposal ŌåÆ `ErrStaleRevision` (409) + "page changed, re-run". ŌåÆ **Block** (never silently overwrite a concurrent edit). |
| **Per-request timeout + token cap** | Every model call | `context.WithTimeout(~60s)` + explicit `MaxTokens` per mode + `MaxStep Ōēż 12` on the ReAct loop. ŌåÆ **Block / fail closed** (structured "provider unreachable / couldn't complete" error, never a silent hang or unbounded loop). |

### Offline (Flywheel)

> 5-user self-hosted scale ŌĆö the flywheel is lightweight: sample the audit log, feed caught failures back into the reference dataset. No streaming analytics platform.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|

## 7. Production Monitoring

> **Honest scale framing:** this is a self-hosted single Go binary for ~5 users. "Production monitoring" here means the **existing `audit.Logger` + structured `slog`**, not a heavyweight observability stack. The richest signal is already captured by design ŌĆö prompt, mode, proposal, and approval/rejection are audited (`ActionAgentPrompt` / `ActionAgentPatchProposal` / `ActionAgentPatchApproval`). Monitoring is reading those, not standing up Phoenix/Grafana.

**Tracing Tool:** Existing **`audit.Logger` (per-action events) + `slog` (structured request logs)**. Phoenix/LangSmith have no Go SDK and a separate runtime contradicts the single-binary deploy; if a visual trace dashboard is ever wanted, emit OpenTelemetry spans (`go.opentelemetry.io/otel`) from the agent service to any OTLP collector ŌĆö **optional, deferred, not MVP**.

**What to log / trace per request** (key never logged):
- Audit event: actor, mode, scope (page/selection/attachment/workspace), target path, and for changes the proposal + approval/rejection decision (already specified).
- `slog`: request id, mode, tool calls made (names + step count), retry count, latency, token usage (DeepSeek cost is cents/day ŌĆö track for drift, not budget), and structured error class on failure (provider-unreachable / step-exhaustion / validation-failed / stale-revision).

**Key Metrics to Track** (lightweight counters/log fields, reviewed not alerted-on at this scale):
1. **Silent-write count** ŌĆö committed `Source="agent"` changes with no matching `ActionAgentPatchApproval`. **Target: 0, always.** This is the one metric that maps to a Critical failure.
2. **Refusal vs. answer rate on out-of-corpus questions** ŌĆö a falling refusal rate is the early warning for creeping fabrication (D2/D7).
3. **Proposal-validation failure / retry-exhaustion rate** ŌĆö how often `validateProposedBody` rejects (fenced body, frontmatter churn) ŌåÆ patch-quality drift (D4).
4. **Tool-call reliability** ŌĆö step-exhaustion rate and malformed-tool-call rate (DeepSeek-specific, ┬¦3 Pitfall 3) (D1).
5. **Stale-revision block rate** ŌĆö confirms D8 fires in the wild (a non-zero rate is healthy ŌĆö it means the guard is catching real concurrent edits, not a failure).
6. **Latency / error rate** ŌĆö p95 turn latency vs. the 60s timeout; provider-unreachable rate.

**Alert Thresholds** (5-user scale ŌåÆ "surface to the maintainer," not pager rotation):
- **Any** silent-write (metric 1 > 0) ŌåÆ Critical, investigate immediately ŌĆö a structural-boundary regression.
- **Any** agent output containing `config`/secret-shaped or out-of-role content ŌåÆ Critical (scope leak).
- Provider-unreachable or step-exhaustion rate spiking ŌåÆ degraded-agent banner in the UI (already specified) + maintainer notice; re-check the DeepSeek model id (deprecation 2026-07-24).
- Proposal-validation-failure rate climbing ŌåÆ patch-quality regression; pull samples into the flywheel.

**Smart Sampling Strategy** (per `ai-evals.md` ŌĆö weight toward concerning signals, not uniform):
- Sample **every** agent-applied commit for the silent-write/approval reconciliation (it's cheap and the highest stakes).
- Weight human review toward: large-diff / high-churn proposals, runbook & decision-record target pages (highest operational stakes), interactions with retries or step-exhaustion, unusually long answers, and any case where a user rejected the diff (a rejection is a labeled negative ŌĆö feed it straight into `agent_eval.jsonl`).
- Watch for **signal-metric divergence**: if users start rejecting diffs more often while the judge still scores groundedness high, the judge has drifted ŌĆö recalibrate against fresh human labels.
