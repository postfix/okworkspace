## Phase Goal

**As a** signed-in teammate, **I want to** have search stay accurate the instant content changes (and have an admin able to rebuild it), **so that** I can trust search to reflect the workspace without ever seeing stale, missing, or ghost results.

<objective>
Harden the search lifecycle so the derived index stays consistent with the files-are-truth source across every mutation, and give admins a manual rebuild. 03-01 built the rebuild + startup-drift backstop; this plan wires the INCREMENTAL layer — a fire-and-forget `KindIndex` job after every page and attachment mutation and after text extraction completes — and surfaces the admin Reindex action in the UI. It also adds the concurrency race test and closes the phase with a single end-of-phase human verification of the full search UX (browser-only checks, per `human_verify_mode: end-of-phase`).

Scope of THIS plan: enqueue hooks at each mutation site, the admin Reindex UI + client mutation, the concurrency race test, and the end-of-phase verification checkpoint. No new search-engine code (that is 03-01/03-03); this plan connects existing mutation paths to the existing `KindIndex` job.

Purpose: Make the live index trustworthy (Pitfall 1 — the phase's primary risk) and complete the admin operational story.
Output: Incremental index enqueues across pages/attachments, the admin Rebuild action, the race test, and a verified end-to-end search experience.
</objective>

<interface_contract>
Reuses, does not extend, the search interfaces:
- `search.KindIndex` job kind + `indexPayload{Op,Kind,Path,ID}` (03-01).
- Helper constructors for payloads (add small `search.UpsertPagePayload(path)`, `search.DeletePagePayload(path)`, `search.UpsertAttachmentPayload(id)`, `search.DeleteAttachmentPayload(id)` to `internal/search` if not already present — thin JSON marshalers mirroring `RebuildPayload()`), so mutation sites enqueue without hand-building JSON.
- `POST /api/v1/admin/search/reindex` (03-01) → 202; the SPA adds `client.reindexSearch()` mirroring the existing CSRF POST helper.

Enqueue-context rule (CRITICAL — CR-01):
- From HTTP-handler goroutines (pages.Create/Save/Rename/Move/Delete, attachments.Upload/Replace/Remove): `worker.Enqueue` (fire-and-forget) is correct AND `EnqueueAndWait` is also permitted (these are NOT the drain goroutine). Use plain `Enqueue` for index jobs (search freshness is an optimization; the rebuild is the correctness backstop) to avoid adding latency to saves/uploads.
- From INSIDE a job handler running on the worker (extractjob.go extraction-done): MUST be `worker.Enqueue` (fire-and-forget) — NEVER `EnqueueAndWait` (deadlocks the single drain goroutine — Phase 2 CR-01).
</interface_contract>

Wire each site AFTER its commit is enqueued/landed (so the index reflects committed state), using `worker.Enqueue` (fire-and-forget):

- `internal/pages/service.go` `Create()` and `Save()`: after the existing `EnqueueCommit` succeeds, `worker.Enqueue(ctx, search.KindIndex, search.UpsertPagePayload(path))`. (HTTP-handler context → fire-and-forget; do not block the save.)

- `internal/pages/trash.go` `Delete()`: after `EnqueueCommit`, enqueue a delete for the trashed page path (removes the page + its heading docs — 03-03 handles heading cleanup in the delete branch). Restore-from-trash (`trash.go` restore path) → enqueue an upsert for the restored path.
- `internal/attachments/service.go` `Upload()` and `Replace()`: after the existing `EnqueueAndWait` commit succeeds, `worker.Enqueue(ctx, search.KindIndex, search.UpsertAttachmentPayload(id))` (indexes filename immediately; extracted text follows when extraction completes).

- `internal/attachments/extractjob.go` extraction-done: AFTER the existing fire-and-forget `w.Enqueue(kindCommit, ...)` of the `.txt` commit, add `w.Enqueue(ctx, search.KindIndex, search.UpsertAttachmentPayload(p.AttachmentID))` — this re-indexes the attachment WITH its extracted text. This runs ON the drain goroutine, so it MUST be `Enqueue`, never `EnqueueAndWait` (CR-01). Mirror the exact fire-and-forget comment style already in extractjob.go. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mutation handler → KindIndex job | Index freshness enqueued across the worker boundary (deadlock risk if mis-contexted) |
| admin browser → POST reindex | Privileged rebuild crosses the admin boundary |
| concurrent HTTP reads ↔ worker writes | Shared bleve.Index accessed by many readers + one writer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-16 | DoS (self-inflicted deadlock) | extractjob.go extraction-done enqueue | mitigate | Fire-and-forget `Enqueue` ONLY inside the handler (CR-01); test asserts no EnqueueAndWait from the drain goroutine. |
| T-03-17 | Information Disclosure | trash delete enqueue | mitigate | Delete-to-trash enqueues an index delete so trashed content leaves results immediately; rebuild backstop reconciles any missed enqueue. |
| T-03-18 | Elevation | admin Rebuild action | mitigate | Endpoint is admin-only + CSRF (03-01); client `reindexSearch` sends the `X-CSRF-Token`; button only shown/usable in the admin route. |
| T-03-19 | Tampering (data race) | shared bleve.Index | mitigate | One shared index, many readers + single worker writer; swap under mutex; `TestIndex_ConcurrentReadWrite` under `-race`. |
| T-03-20 | Availability | incremental enqueue failure | accept | Index freshness is best-effort; a dropped enqueue is reconciled by startup drift rebuild + admin rebuild (rebuild is the correctness layer). Logged at Warn. |
</threat_model>
