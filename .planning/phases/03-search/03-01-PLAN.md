## Phase Goal

**As a** signed-in teammate, **I want to** search page titles, body text, and tags and get typed results, **so that** I can quickly find any knowledge in the workspace without remembering where it lives.

> NOTE: The ROADMAP `**Goal:**` line for Phase 3 is in outcome prose ("A user can quickly find any knowledge..."), not strict user-story form. The story above is a faithful rendering of that exact goal and its success criteria — no scope was invented. If a stricter story is desired, run `/gsd mvp-phase 3`.

<objective>
Stand up the backend search foundation as the first end-to-end vertical slice: a user can issue an authenticated query and get typed, ranked PAGE results (title / body / tag). This is the walking-skeleton equivalent for search — it wires the locked Bleve v2.6.0 engine into the existing single-binary architecture and proves the full read path (HTTP → query builder → on-disk index → typed JSON) plus the correctness backstop (idempotent rebuild-from-files + startup drift detection).

Scope of THIS plan: pages only (title/body/tag → SRCH-01/02/03), the `type` field + facet for typed results (SRCH-06, page type), the rebuild + drift lifecycle, and the authed/admin endpoints. Attachment indexing, extracted-text, heading deep-links, the ⌘K UI, and incremental mutation-hook wiring are SEPARATE plans (03-02, 03-03, 03-04).

Purpose: Make the locked Bleve stack implementable and prove the end-to-end search path with the rebuild safety net that is this phase's primary engineering risk.
Output: New `internal/search` package, `gitstore.HeadSHA`, migration 0007, `GET /search` + `POST /admin/search/reindex`, startup wiring, and the Wave 0 test scaffolds for the whole phase.
</objective>

<interface_contract>
Result JSON shape (returned by GET /search, consumed by 03-02 SearchPalette). The DTO is defined here and is the contract for the frontend:

- Each result object: `kind` ("page"|"heading"|"attachment"), `title` (string), `path` (workspace-relative page path to navigate to), `snippet` (weight-only-safe highlighted fragment string, no `<mark>`), `anchor` (optional, heading only), `page_title` (optional, heading/attachment only).
- Top-level response: a JSON array of result objects. Empty query returns `[]` with 200 (fast path, no Bleve call).
- THIS plan emits only `kind:"page"` results; `heading`/`attachment` kinds are added by 03-03 without changing the envelope.

Go-side identifiers this plan defines (consumed by 03-03):
- Constants `search.TypePage`, `search.TypeHeading`, `search.TypeAttachment`, `search.KindIndex`.
- `search.IndexHandler(idx *search.Index, r *repo.Repo) jobs.Handler`.
- `search.OpenOrCreate(dir string) (*search.Index, error)` (or `Open`), `(*Index).Close()`, `(*Index).Query(ctx, q string) ([]Result, error)`, `(*Index).RebuildIndex(ctx) error`, `(*Index).DriftCheck(ctx, gs) (bool, error)`, `search.RebuildPayload() string`.
- `indexPayload{ Op, Kind, Path, ID string }` with `Op` in {"upsert","delete","rebuild"}.
</interface_contract>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → GET /search | Untrusted `q` query string crosses into the Bleve query builder |
| browser → POST /admin/search/reindex | Privileged operational action crosses the admin boundary |
| files on disk → index | Page bytes (already trusted content) read via the safe-path resolver into the derived index |
| index dir ↔ Git repo | Derived index must never enter the content/Git repo |
