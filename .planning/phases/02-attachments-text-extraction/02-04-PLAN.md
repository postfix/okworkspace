## Phase Goal

**As an** internal team member (editor), **I want to** replace an attachment with a new version and remove an attachment's link from a page — with unreferenced files cleaned up automatically — **so that** pages stay tidy and storage doesn't accumulate orphaned files, while old versions remain recoverable.

> Slice 4 of 4 — completes the phase. Builds on 02-01/02/03. After this plan the full attachment lifecycle (upload, preview, extract, replace, remove, orphan-delete) is in place.

<objective>
Add attachment lifecycle management: Replace (ATT-05 — reuse the same opaque id, write new bytes + updated meta through the single-writer CommitJob so history retains the prior version, then re-enqueue extraction), Remove link (ATT-06 — drop the canonical link from the page markdown via the existing page-save path), and orphan deletion (ATT-07 — when the removed link was the last reference across ALL pages, delete the binary + JSON + TXT sidecars in ONE commit). Surface Replace/Remove as editor-only card actions behind focus-trapped confirm dialogs.

Purpose: Close the lifecycle loop. Editors can manage attachments safely; storage never accumulates orphans; nothing referenced elsewhere is ever deleted. The canonical link form defined in 02-01 (`DownloadRefPath(id)`) is the single match point so insert and scan can never drift (Pitfall 6).

Output: Replace/Remove/orphan-delete on `attachments.Service`, the full `PageReferences` scan in `refs.go`, the replace/delete HTTP handlers + routes, and the `ReplaceAttachmentDialog`/`RemoveAttachmentDialog` UI wired into the card.
</objective>

<interface_contracts>
**Canonical reference (from 02-01, reused here):** `attachments.DownloadRefPath(id)` → `/api/v1/attachments/<id>/download`. `PageReferences` matches this exact substring across all page markdown; the unlink edit and the upload insert both use it. They MUST never diverge (Pitfall 6).

**Service methods:**
- `Replace(ctx, id, filename string, data []byte, user string) (AttachmentMeta, error)` — reuse the SAME id; MIME-sniff + size-cap the new bytes (same validation as Upload); write new bin (same ext path; if the new sniffed ext differs, write the new path AND add the old path to Removes in the same commit) + updated meta; re-enqueue KindExtract. Hidden-Git: history retains the prior version automatically.
- `Remove(ctx, id, pagePath, user string) (deletedOrphan bool, error)` — (1) edit `pagePath`'s markdown to drop the canonical link via the existing page-save/commit path (ATT-06); (2) `refs.PageReferences(ctx, id)` across ALL pages; (3) if count == 0, build ONE commitPayload with `Removes: [BinPath, MetaPath, TxtPath]` (TXT only if it exists) and delete the attachments row (ATT-07); else keep the files. Mirror the multi-file commit batching in `internal/pages/trash.go`.

**RBAC + routes:** `PUT /attachments/{id}` (replace) and `DELETE /attachments/{id}` (remove) under the existing `editor` subgroup; authorization from the session via RequireRole (never client input).
</interface_contracts>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → replace/delete API | Editor-initiated mutations with attacker-shaped ids/page paths |
| Unlink edit → orphan decision | A reference-scan miss could delete a still-used file or strand an unused one |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-14 | Elevation | PUT/DELETE /attachments/{id} | mitigate | Both routes under the editor subgroup; `RequireRole(editor)` from the session, never client input; CSRF on mutating methods (existing nosurf) |
| T-02-15 | Tampering (wrongful delete / strand) | Remove orphan ref-count | mitigate | `PageReferences` scans ALL page markdown for the single canonical `DownloadRefPath(id)`; delete only when count==0, in ONE commit (Pitfall 6); insert+scan share the helper so they cannot drift |
| T-02-16 | DoS | handleReplaceAttachment | mitigate | `http.MaxBytesReader` on the replace multipart body before parsing (same cap as upload); re-sniff type vs allow-list |
| T-02-17 | Tampering (path traversal via id/pagePath) | Replace/Remove paths | mitigate | id is opaque (validated non-empty/NUL-free); all paths via `repo.Resolve` (SEC-01); attachment paths built only from the server id, never the filename |
| T-02-SC | Tampering | dependency installs | accept | No new dependencies in this slice (all deps pinned in 02-01) |
</threat_model>
