## Phase Goal

**As an** internal team member (editor), **I want to** attach original files to a page, download them byte-for-byte unchanged, and manage them safely with their text extracted, **so that** knowledge stays as portable open files that search and the agent can later read.

> This is the first of four vertical slices. After this plan, the end-to-end upload → store → download path works for real (real commit, real byte-exact download, real validation). Slices 2–4 add image preview/card detail (02-02), text extraction + status (02-03), and replace/remove/orphan-delete (02-04).

<objective>
Deliver the thinnest real end-to-end attachment slice: an editor drops a file into a page's Attachments section, the server caps + MIME-sniffs it, assigns an opaque ULID id, writes the binary + JSON meta sidecar through the existing single-writer CommitJob (one commit), and the file is downloadable byte-for-byte with the SEC-02 disposition. Also lay the package foundation (deps, fixtures, config, migration, types/meta/id/refs) that slices 02-02/02-03/02-04 build on.

Purpose: Prove the full upload→commit→download stack before adding card detail, extraction, and lifecycle. Establishes the `internal/attachments` package mirroring `internal/pages`, with the hard ATT-02 byte-exact guarantee and ATT-09/SEC-02 security boundary in place from task one.

Output: `internal/attachments` package (service, id, meta, refs, types), the multipart upload + byte-exact download handlers, the attachments table migration, the four Go deps pinned, extract fixtures, and a minimal editor-gated Attachments UI (dropzone + filename card + download) mounted in PageView.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → upload API | Untrusted multipart body: arbitrary size, attacker-controlled filename + bytes |
| Stored bytes → download response | Stored binary streamed back to a browser that may render it |
| Filename/extension → on-disk path | Client-supplied name must never influence the storage path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | DoS | handleUploadAttachment | mitigate | `http.MaxBytesReader(r.Body, max_upload_mb)` BEFORE ParseMultipartForm caps the body before buffering/spooling (Pitfall 1); return 413 |
| T-02-02 | Spoofing/Tampering | Upload type check | mitigate | `mimetype.Detect` magic-byte sniff vs `config.AllowedExtensions`; reject sniffed-type mismatch (ATT-09); never trust hdr.Filename |
| T-02-03 | Tampering (path traversal) | on-disk path | mitigate | Server ULID path only; original filename kept solely in meta; `repo.Resolve` (SEC-01) on every path |
| T-02-04 | Tampering/Elevation (stored XSS) | Download/preview of SVG/HTML | mitigate | Non-images: `Content-Disposition: attachment` + `X-Content-Type-Options: nosniff` (SEC-02); images served to `<img>` only; explicit Content-Type per sniffed type |
| T-02-05 | Elevation | POST /attachments | mitigate | Registered under the editor subgroup; `RequireRole(editor)` authorizes from the session, not client input |
| T-02-06 | Tampering (byte mutation) | Download path | mitigate | `http.ServeContent`/no transcode; extraction (later slice) reads a copy, never writes the binary path (ATT-02/Pitfall 4) |
| T-02-SC | Tampering | Go module installs | accept | Go modules verified via Go proxy in RESEARCH Package Legitimacy Audit (ledongthuc/pdf, fumiama/go-docx, mimetype, oklog/ulid — all OK, non-archived, pinned in go.sum). Go modules are outside the npm/pip/cargo legitimacy seam; pseudo-versions committed to lockfile. |
</threat_model>
