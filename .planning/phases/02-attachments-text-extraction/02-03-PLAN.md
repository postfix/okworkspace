## Phase Goal

**As an** internal team member, **I want to** have text automatically extracted from my PDF/DOCX/TXT attachments with a live, honest status, **so that** search and the agent can read them and I can tell when a file (e.g. a scanned PDF) simply has no readable text.

> Slice 3 of 4. Builds on 02-01/02-02 (upload + cards exist). After this plan, extractable uploads get a `<id>.txt` sidecar and a live status chip with an explicit, non-alarming "No text extracted" state. Slice 02-04 adds replace/remove/orphan-delete.

<objective>
Add asynchronous text extraction (ATT-08): a `KindExtract` job on the existing single jobs worker reads the committed binary, extracts text via pure-Go `ledongthuc/pdf` (PDF) / `fumiama/go-docx` (DOCX) / stdlib (TXT), and commits an `<id>.txt` sidecar through the same single-writer CommitJob. Surface the lifecycle to the card via an SSE endpoint and an `ExtractionStatus` chip modeled on `AutosaveStatus`. Distinguish the empty-but-succeeded case ("No text extracted", e.g. scanned PDF) from a genuine parse failure ("Couldn't extract text"), and guarantee a third-party parser panic cannot kill the drain goroutine.

Purpose: Make attachment text available to Phase 3 (search) and Phase 4 (agent) while giving the user an honest, live status. The "No text extracted" path must be solid — it is the legitimate result for scanned/image PDFs (text-layer only).

Output: `extract.go` (pure extractors + tests against fixtures), `extractjob.go` (KindExtract handler), `status.go` (extraction-status source), the SSE handler + route, the `ExtractionStatus` chip wired into the card via `EventSource`, and the worker registration in main.go.
</objective>

<interface_contracts>
**Error-handling contract (drives the four UI states — RESEARCH §Extraction-Fidelity):**
| Outcome | ExtractJob result | `<id>.txt` | status | chip |
|---------|-------------------|-----------|--------|------|
| Text found | success | text bytes | done | "Text extracted" (green) |
| Empty (scanned PDF / blank doc) | **success** | empty file | empty | "No text extracted" (amber) + sub-note |
| Parse error (corrupt/encrypted) | error → retry → terminal fail | not written | failed | "Couldn't extract text" (red) |
| Non-extractable type (image/other) | no ExtractJob enqueued | none | n/a | no chip |

Write a (possibly empty) `<id>.txt` on success so the card distinguishes empty-but-succeeded from not-yet-run. Reserve "failed" for genuine parse errors after retry exhaustion. `recover()` inside the handler AND each extractX converts a panic into a returned error so one bad file never kills the single drain goroutine (Pitfall 5).

**Status source of truth:** the `attachments` table `extract_status` column (pending/done/empty/failed) + `extract_error`. The ExtractJob updates this row; the SSE endpoint reads it (survives job-row pruning — simpler than reading the jobs table). Define `status.go` `ExtractionStatusFor(ctx, id) (ExtractionStatus, error)` reading that column.

**SSE shape:** `GET /api/v1/attachments/{id}/status` → `text/event-stream`, each event `data: {"status":"extracting|done|empty|failed"}\n\n`; close the stream on a terminal status or `r.Context().Done()`. EXEMPT this route from any global write timeout (Pitfall 7) — set a generous/no per-connection write deadline so the long-lived stream is not killed.
</interface_contracts>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Stored binary → parser | Adversarial/malformed PDF/DOCX fed to a third-party parser inside the single job-drain goroutine |
| Job status → SSE stream | Long-lived response to an authenticated browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | DoS (drain-goroutine kill) | ExtractHandler / extractX | mitigate | `defer recover()` inside each extractor AND the handler converts a parser panic into a returned error so the single worker goroutine survives (Pitfall 5) |
| T-02-10 | DoS (silent stream death) | handleExtractionStatus | mitigate | Exempt SSE route from global write timeout; close on terminal status or context cancel; UI degrades gracefully to last-known state (Pitfall 7) |
| T-02-11 | Tampering (binary mutation) | ExtractHandler | mitigate | Extractor reads a COPY of the bytes; only `<id>.txt` is ever written — the original binary path is never written by extraction (ATT-02/Pitfall 4) |
| T-02-12 | Information disclosure | extract_error column / failed chip | mitigate | The failed chip shows the fixed copy "Couldn't extract text"; raw parser error strings stay server-side (extract_error), never sent to the client |
| T-02-13 | Spoofing/Elevation | SSE endpoint | mitigate | Route under the authed group (loadCurrentUser); any authenticated user may read status (same as reading a page); not editor-gated by design |
| T-02-SC | Tampering | dependency installs | accept | No new dependencies (pdf/docx/mimetype/ulid pinned in 02-01; native EventSource is a browser standard) |
</threat_model>
