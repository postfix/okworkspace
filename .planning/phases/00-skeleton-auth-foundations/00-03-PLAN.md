<objective>
Deliver role-based access control and the team-management surface that exercises it. Build the `RequireRole(admin|editor|reader)` middleware (server-side authorization on every protected action, never client-trusted), the in-app `/admin` user-management screen + API (list / add / set-role / reset-password / deactivate, admin-only â€” D-05), self-service profile (own password + display name, but not own role â€” D-06), the forced first-login password-change flow (D-02), the logout control reachable from any page (AUTH-02), and the CLI `admin reset-password` lockout-recovery subcommand (D-04).

Purpose: AUTH-05 (role gating) and AUTH-02 (logout) plus the team-onboarding surface a non-technical admin needs (CONTEXT.md core value). In Phase 0 the role-gated surface that fully exercises RBAC is admin-only access to `/admin`; editor-vs-reader content gating becomes meaningful in Phase 1, but the roles + `RequireRole` middleware must be built and enforced now (D-07).
Output: A working RBAC boundary, an onboarding-ready admin screen, self-service profile, enforced first-login password reset, and a CLI recovery path.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated user -> admin-only API | A non-admin must not perform user-management actions |
| Client request -> server authorization | The client must never be trusted to assert its own role |
| User -> own account vs others' accounts | A user may edit only their own profile/password, never their own role or others' accounts |
| Operator shell -> CLI password reset | The CLI reset shares the bootstrap trust boundary (requires shell access to the box) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00.03-01 | Elevation of privilege | admin-only routes | mitigate | `RequireRole(RoleAdmin)` middleware reads the role from the session-bound user (never client input); 403 for editor/reader, 401 for unauthenticated; AUTH-05. |
| T-00.03-02 | Elevation of privilege | self-service profile | mitigate | `UpdateOwnProfile`/`ChangeOwnPassword` accept NO role parameter and operate only on the session user's own id; a user cannot escalate their own role (D-06). |
| T-00.03-03 | Tampering / CSRF | admin + profile mutations | mitigate | All mutating admin/profile routes behind nosurf CSRF (SEC-04, established Plan 01) in addition to RBAC. |
| T-00.03-04 | Elevation of privilege | forced password change bypass | mitigate | must_change_password gates the app server-side (the flag is checked on /auth/me and protected routes), not only hidden in the UI; D-02. |
| T-00.03-05 | Spoofing | password reset credential | mitigate | Reset generates a strong one-time password (Argon2id-hashed), printed once, with must_change_password=1; never a fixed/default value; D-04. |
| T-00.03-06 | Denial of service | deactivated account still usable | mitigate | `Deactivate` sets active=0 and `Authenticate` rejects inactive users so a disabled teammate cannot sign in. |
| T-00.03-07 | Information disclosure | one-time passwords in logs | accept | CLI/admin reset prints the one-time password to the operator log by design (D-04 recovery path, same trust boundary as bootstrap); login-path plaintext is still never logged. |
| T-00.03-SC | Tampering (supply chain) | new npm/go deps | mitigate | No new packages beyond the locked list (UI uses the Plan-01 hand-built primitives + lucide-react already locked); if any new dep is required it triggers a blocking legitimacy checkpoint before install. |
</threat_model>
