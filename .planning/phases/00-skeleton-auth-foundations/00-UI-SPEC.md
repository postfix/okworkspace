# Phase 0 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** Phase 0 is a Walking Skeleton. It establishes the *foundational* design tokens (spacing, type scale, color, radius/elevation) that every later phase inherits, and applies them to the only surfaces this phase delivers: the login screen, the forced first-login password change, the authenticated AppShell (top bar + left nav rail + main pane placeholder), the admin user-management screen, and the self-service profile. It does NOT style page editing, Markdown rendering, attachments, search, or the agent — those arrive Phase 1+.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual token system) |
| Preset | not applicable |
| Component library | none — hand-built primitives (Button, Input, Field, Table, Dialog, Banner) styled with plain CSS + CSS custom properties |
| Icon library | `lucide-react` (tree-shakeable, MIT, no runtime deps) — only icons actually used are imported |
| Font | System font stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif`) for UI; `ui-monospace, SFMono-Regular, "SF Mono", Menlo, Consolas, monospace` reserved for code/IDs. No web-font download — keeps the single-binary deploy lean and avoids FOUT. |

**Why no shadcn (documented decision):** The locked frontend stack (CLAUDE.md / STACK.md) is React 19 + Vite 8 + TS 6 with `@uiw/react-md-editor`, `react-markdown` + `rehype-sanitize`, `@tanstack/react-query`, `zustand`, `react-router-dom`. It deliberately does **not** include Tailwind or Radix. shadcn would pull in Tailwind + Radix dependencies outside the locked list, contradict the project's "minimal deps / single static binary" constraint, and add a build surface the team did not choose. The sensible default for this greenfield phase is a small, hand-built token system using CSS custom properties (`web/src/styles/tokens.css`) and a handful of unstyled-then-themed primitives. Every token below maps 1:1 to a CSS variable so later phases (and an eventual editor theme) inherit one source of truth. shadcn can be reconsidered in a future milestone if the component surface grows, but it is out of scope here. Registry safety gate is therefore **not applicable**.

**Token delivery:** Declare all tokens as CSS custom properties on `:root` in `web/src/styles/tokens.css`, imported once at the app entry. Components reference `var(--…)` only — no hard-coded pixel/hex values in component CSS.

## Spacing Scale

8-point scale (all multiples of 4). Declare as `--space-*` custom properties.

| Token | Value | Usage |
|-------|-------|-------|
| `--space-xs` | 4px | Icon-to-label gaps, inline padding, focus-ring offset |
| `--space-sm` | 8px | Compact element spacing, form-field internal padding, table cell padding-y |
| `--space-md` | 16px | Default element spacing, gap between form fields, card/dialog padding |
| `--space-lg` | 24px | Section padding, login card inner padding, main-pane padding |
| `--space-xl` | 32px | Layout gaps, vertical rhythm between major blocks |
| `--space-2xl` | 48px | Major section breaks, empty-state vertical centering offset |
| `--space-3xl` | 64px | Page-level spacing, login screen top offset |

**Layout fixed dimensions (phase-specific, declared as tokens):**

| Token | Value | Usage |
|-------|-------|-------|
| `--topbar-height` | 56px | Authenticated AppShell top bar height |
| `--navrail-width` | 264px | Left navigation rail width (expanded) |
| `--login-card-width` | 400px | Max width of the centered login / password-change card |

Exceptions: Interactive controls (buttons, inputs, nav rows, table action buttons) must meet a **minimum 40px hit height** (44px for icon-only buttons) for non-technical-user accessibility, even when visual padding would compute smaller. This is the only allowed deviation from the 4-multiple spacing rule and applies to target sizing, not layout gaps.

## Typography

Exactly 4 sizes, exactly 2 weights. Declare as `--font-size-*`, `--font-weight-*`, `--line-height-*`.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (login wordmark, screen H1) | 28px | 600 (semibold) | 1.2 |
| Heading (section titles, dialog titles, admin "Users" header) | 20px | 600 (semibold) | 1.2 |
| Body (default text, inputs, table cells, nav labels, button text) | 16px | 400 (regular) | 1.5 |
| Label / caption (field labels, helper text, table column headers, role badges, audit-quiet meta) | 14px | 400 (regular) — uppercase tracking only on table column headers | 1.4 |

- Weights: **only** 400 (regular) and 600 (semibold). No light, medium, or bold variants.
- Sizes: **only** 28 / 20 / 16 / 14. No 12px text (too small for non-technical readers); use 14px as the smallest.
- Body line-height 1.5; heading/display line-height 1.2; label line-height 1.4.
- Numerals in tables/IDs may use `font-variant-numeric: tabular-nums` for alignment (presentation-only, not a new size/weight).

## Color

60 / 30 / 10 split. Light theme is the Phase 0 baseline; tokens are named semantically so a dark theme can be added later without renaming. Declare as `--color-*`.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) — `--color-bg` | `#FFFFFF` | App background, main content pane, input fields |
| Dominant text — `--color-text` | `#1A1A1A` | Primary body text on dominant surface |
| Secondary (30%) — `--color-surface` | `#F6F7F8` | Left nav rail, top bar, admin table header row, login card backdrop |
| Secondary border — `--color-border` | `#E3E5E8` | Hairline borders, dividers, table row separators, input borders |
| Muted text — `--color-text-muted` | `#6B7280` | Helper text, table meta (size/date), placeholder, disabled-tree "editing arrives next" copy |
| Accent (10%) — `--color-accent` | `#2563EB` | See reserved list below |
| Accent text-on-accent — `--color-accent-fg` | `#FFFFFF` | Text/icon on accent fills |
| Destructive — `--color-destructive` | `#DC2626` | See reserved list below |
| Success (status only) — `--color-success` | `#16A34A` | Repo-health "OK" dot, "password changed" confirmation only |
| Warning (status only) — `--color-warning` | `#D97706` | Stale-lock self-heal notice, Git-divergence "refused to auto-merge" alert, "must change password" banner |

**Accent (`#2563EB`) reserved exclusively for:**
1. The single **primary button** per screen (Login "Sign in"; password-change "Update password"; admin add-user "Create user").
2. The **active/current** nav-rail row indicator (left bar + label color) — even though the tree is read-only this phase, the selected node uses accent.
3. Text links (logout is a button, not a link, but any inline link uses accent).
4. Input **focus ring**.

Accent is **never** used for: every interactive element, all icons, decorative fills, the wordmark, table rows, or borders.

**Destructive (`#DC2626`) reserved exclusively for:**
1. The "Deactivate user" / "Disable user" action in the admin table (and its confirmation dialog's confirm button).
2. The destructive-confirmation dialog accent (title icon + confirm button).

Success/Warning are **status-signal only** (dots, banners) — never used as button fills or large surfaces, so they do not break the 60/30/10 budget.

**Contrast:** All text/background pairs must meet WCAG AA (4.5:1 for body, 3:1 for large/UI). `#1A1A1A` on `#FFFFFF` and `#FFFFFF` on `#2563EB` pass; `--color-text-muted` (`#6B7280`) on `#FFFFFF` passes AA for body (4.6:1).

**Radius & elevation (foundational tokens):**

| Token | Value | Usage |
|-------|-------|-------|
| `--radius-sm` | 6px | Inputs, buttons, role badges |
| `--radius-md` | 10px | Cards, dialogs, table container |
| `--shadow-card` | `0 1px 2px rgba(16,24,40,0.06), 0 1px 3px rgba(16,24,40,0.10)` | Login card, dialog |
| `--shadow-popover` | `0 4px 12px rgba(16,24,40,0.12)` | Profile/user dropdown menu in top bar |

## Copywriting Contract

Plain, calm, non-technical voice (the audience includes non-technical teammates). Sentence case, no jargon, no exclamation marks. Errors say what happened and what to do next.

| Element | Copy |
|---------|------|
| Primary CTA — Login | **Sign in** |
| Primary CTA — Forced password change | **Update password** |
| Primary CTA — Admin add user | **Create user** |
| Primary CTA — Profile save | **Save profile** |
| Login screen title | **OKF Workspace** (wordmark) with subtitle "Sign in to your workspace" |
| Login fields | "Username", "Password" (labels above fields) |
| Forced-password-change banner | "You're using a temporary password. Choose a new one to continue." |
| Forced-password-change title | "Set a new password" |
| Empty state heading — main pane (Phase 0 placeholder) | **Your workspace is ready** |
| Empty state body — main pane | "Page editing arrives in the next release. For now you can browse the starter structure on the left. Your admin can add teammates from the admin screen." (admins see "…from the admin screen." → linked; non-admins omit the admin sentence.) |
| Empty state heading — admin users (only if zero non-admin users) | **No teammates yet** |
| Empty state body — admin users | "Add your team so they can sign in. Use Create user to get started." |
| Error state — login failure | "Invalid username or password." (generic, identical for wrong-user and wrong-password — no account enumeration) |
| Error state — network/server | "Something went wrong. Check your connection and try again." |
| Error state — session expired | "Your session expired. Sign in again to continue." (redirect to /login) |
| Error state — password too weak | "Choose a longer password — at least 12 characters." |
| Error state — new passwords don't match | "The two passwords don't match." |
| Error state — admin action forbidden (RBAC) | "You don't have permission to do that." |
| Warning banner — stale Git lock self-healed | "Recovered from an interrupted save. Everything looks fine." (admin-visible repo-health area) |
| Warning banner — Git remote divergence | "The remote repository has diverged. Automatic sync was paused to protect your data — contact your administrator." |
| Status — repo health OK | "Storage healthy" (with success dot) |
| Logout control label | **Log out** (in the top-bar user menu) |
| Destructive confirmation — Deactivate user | Title: "Deactivate {displayName}?" · Body: "They won't be able to sign in until reactivated. Their account and history are kept." · Confirm button: "Deactivate" (destructive) · Cancel: "Keep account active" |
| Destructive confirmation — Reset user password | Title: "Reset password for {displayName}?" · Body: "This generates a new temporary password they must change on next sign-in." · Confirm: "Reset password" · Cancel: "Keep current password" |

**Destructive actions in this phase (each requires an explicit confirmation dialog — never a one-click destructive action):**
1. **Deactivate / disable a user** (admin table row) → confirmation dialog above.
2. **Reset a user's password** (admin table row) → confirmation dialog above (treated as semi-destructive because it invalidates the current credential).

There is **no page delete, no content deletion, and no Git-history mutation** in Phase 0, so those destructive flows are out of scope here.

## Visual Focal Point (per Phase-0 surface)

Each surface declares the single element that anchors attention, so the executor builds a clear visual hierarchy (one focal point per screen; everything else recedes).

| Surface | Visual focal point |
|---------|-------------------|
| Login (`/login`) | Centered login card; the **OKF Workspace wordmark** is the highest-weight element (28px/600), and the **"Sign in"** primary CTA (accent fill) is the action anchor — everything else (fields, subtitle) is secondary. |
| Forced password change | Same centered card; the **"Set a new password"** heading anchors, the **"Update password"** primary CTA is the action anchor; the temporary-password warning banner is supporting, not focal. |
| AppShell (`/app`) | The **main content pane** (empty-state heading "Your workspace is ready") is the focal point; the left **nav rail and top bar are secondary/recessive** (secondary surface, muted, read-only tree). |
| Admin users (`/admin`) | The **users table** is the focal point; the **"Create user"** primary CTA sits top-right as the action anchor; row actions and the table header are secondary. |
| Profile | The **profile form** (display name + password fields) is the focal point; the **"Save profile"** primary CTA is the action anchor. |

## Component Inventory (Phase 0 surfaces)

Primitives to build once (token-driven), reused everywhere:

| Component | Used by | Key states |
|-----------|---------|-----------|
| `Button` (variants: primary/secondary/destructive/ghost) | Login, password change, admin, profile, top bar | default, hover, active, focus-visible, disabled, loading |
| `Input` + `Field` (label + helper/error) | Login, password change, add-user, profile | default, focus, error, disabled |
| `Banner` (info/warning/success) | Forced-password notice, repo divergence, self-heal, save confirmation | one per type |
| `TopBar` | AppShell | shows wordmark, repo-health dot, user menu (display name → Profile, Log out) |
| `NavRail` | AppShell | renders seeded tree **read-only/disabled**; active node uses accent; non-interactive rows are muted |
| `MainPanePlaceholder` | AppShell | empty-state copy above |
| `Table` + row actions | AdminSettings users list | columns: Display name, Username, Role (badge), Status; row actions: Reset password, Deactivate |
| `Dialog` (modal) | Add user, destructive confirmations | focus-trapped, Esc-to-cancel, confirm/cancel |
| `RoleBadge` | Admin table | admin / editor / reader (neutral surface badge, not accent) |
| `UserMenu` (popover) | TopBar | Profile, Log out |

**Routes styled this phase (SPEC §18.1):** `/login`, `/login` → forced password-change view, `/app` (AppShell + placeholder), `/admin` (admin-only, RBAC-gated), profile (within AppShell / dialog).

**Interaction contracts:**
- Every interactive element has a visible **focus-visible** ring (accent, 2px, 2px offset) — keyboard operable for accessibility.
- Loading buttons show an inline spinner and disable; never a layout shift.
- Dialogs trap focus, close on Esc and on backdrop click (except destructive dialogs: backdrop click does **not** confirm — only the explicit confirm button does).
- The logout control is reachable from **any** authenticated page via the persistent top bar (AUTH-02).
- Disabled nav-tree rows must look clearly non-interactive (muted text, no hover affordance, `cursor: default`) so users understand editing is not yet available.
